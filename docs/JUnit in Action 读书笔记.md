# 读书笔记 | 《JUnit in Action》

## 第一部分：JUnit 基础

### 第一章 JUnit 跳门槛

这一章从一个问题出发：谁来执行测试？

作者描述了程序员每天都在做的事：写代码、编译、运行、点击按钮看结果。这套「程序员验收测试」在小规模时是可行的，但随着系统增大，手工测试的问题就暴露了——重复劳动、容易遗漏、无法批量运行。

JUnit 的出现就是为了解决这个问题。它不是唯一的单元测试框架，但它是 Java 世界里事实上的标准。

第一章还覆盖了 JUnit 的安装（当时还是 JAR 包加 classpath 的方式）以及第一个测试类的编写。如果你是从 TDD 那本书过来的，会发现这里的写法更接近传统模式——先写产品代码，再写测试代码——而不是 TDD 的「先写测试再让测试通过」。

### 第二章 探索 JUnit 核心

这一章是全书工具属性的集中体现，几乎所有 JUnit 核心 API 都在这里。

**TestCase 和 TestSuite**：JUnit 4 里一个测试类继承 TestCase，测试方法以 test 开头。TestSuite 是测试运行器，它能自动发现并运行一个或多个 TestCase 类。这是 JUnit 4 最基本的组织单元。

**生命周期**：setUp() 和 tearDown() 分别在每个测试方法运行前后执行，用于初始化和清理测试环境。这两个方法是 JUnit 4 里使用频率最高的生命周期钩子。

**断言**：assertEquals、assertTrue、assertFalse、assertNotNull、assertNull、assertSame、fail() 等构成了 JUnit 的断言体系。其中 assertEquals 使用equals()方法比较两个对象，这是最容易踩坑的地方——如果被比较的对象的equals()方法没有正确定义，断言可能误通过。

**异常测试**：有两种写法。一种是 try-catch 加 fail()——在try块里执行预期抛异常的代码，然后用fail()来fail如果没有抛出异常。另一种是通过@Test(expected=SomeException.class)注解声明预期异常，后者写法更简洁，但失去了对异常消息的验证能力。

**测试运行方式**：除了通过 IDE 运行，JUnit 还可以通过文本界面运行——java org.junit.runner.JUnitCore TestClassName。这在自动化脚本里很有用。

### 第三章 掌握 JUnit

这一章用一个实际案例把前两章的内容串联起来：实现一个基于 Controller 设计模式的简单应用，然后用 JUnit 测试它的每一个部分。

Controller 模式的核心思想是让请求处理和业务逻辑分离。书中以一个典型的 Java EE 场景为例：一个前端控制器接收请求，分发到具体的 Action 类处理，每个 Action 类里有测试需要覆盖的业务逻辑。

这一章还讨论了 JUnit 的一些最佳实践：测试的命名（用描述性的方法名说清楚测了什么）、测试的组织（按被测类还是按功能组织）、测试之间的隔离（每个测试独立运行，不依赖其他测试的执行顺序或结果）。

### 第四章 软件测试原则

这一章跳出 JUnit 本身，思考为什么需要测试。

作者引用了 Guy Kawasaki 的一句俏皮话：「崩溃是竞争对手的程序死了。你的程序死了，叫'特性'。」然后话锋一转—— crashes 之后的 ID 02 消息背后的意思是：这个产品本来还应该多测几个月。

软件测试的核心价值在于：

- **更大的测试覆盖率**：更多的测试意味着更少的漏网 bug
- **提升团队生产力**：自动化测试替代手工回归，开发者把时间花在真正需要思考的地方
- **及早发现回归**：bug 发现得越晚，修复成本越高
- **有信心的重构**：没有测试的重构是盲目的
- **记录预期行为**：测试代码是比注释更可靠的文档

作者把软件测试分成几类：**单元测试**（Unit testing）针对单个类或方法；**集成测试**（Integration testing）验证组件之间的协作；**系统测试**（System testing）验证整个系统；**验收测试**（Acceptance testing）从用户角度出发验证系统是否符合需求。

理解这些测试类型之间的关系是这一章最重要的收获——单元测试是金字塔的底层，数量最多、成本最低；验收测试在顶层，数量少但最接近用户真实需求。

### 第五章 测试覆盖率和开发

这一章探讨两个相关联的问题：如何衡量测试的质量，以及如何写出容易被测试的代码。

测试覆盖率工具（如 EMMA、JaCoCo）通过字节码插桩来跟踪哪些代码被执行了。但覆盖率只能回答「哪些代码被跑了」，不能回答「跑的行为对不对」。100% 的覆盖率不等于高质量的测试。

作者在这一章引入了可测试性（Testability）的概念：什么样的代码容易测试？

核心原则包括：减少不必要的依赖、提供简单的构造函数、避免在构造函数或静态初始化块里做过重的操作、遵循迪米特法则（一个对象只了解它直接交互的对象）。这些都是从实际教训里总结出来的经验。

最后简单提到了测试驱动开发（TDD），但这一章没有深入——毕竟 TDD 的书你已经读过一本了。

## 第二部分：测试策略

### 第六章 使用 Stub 进行粗粒度测试

从这一章开始进入测试策略的具体技术。

Stub 的核心思想是：给依赖的外部组件打桩，返回预设的响应，从而隔离被测代码和外部环境。如果你要测试一段 JDBC 代码，但不连接真实数据库，就可以用 Stub 来模拟 Connection、Statement、ResultSet 的行为。

这一章的重点是用嵌入式 Jetty 服务器模拟 HTTP 服务器。嵌入式服务器的好处是测试可以在本地运行，不依赖外部网络，而且可以模拟各种网络异常场景。Stub 适合的场景是：接口行为已知且稳定，只需要验证被测代码能否正确处理各种响应。

### 第七章 使用 Mock 对象

Mock 和 Stub 的区别是这一章的核心议题。

Stub 模拟的是「返回什么」，Mock 除了模拟返回值，还验证「方法是否以预期的方式被调用」——调用顺序、次数、参数值。Stub 回答「结果对不对」，Mock 回答「过程对不对」。

Tim Mackinnon、Steve Freeman 和 Philip Craig 在 XP2000 上首次提出了 Mock 对象这个概念。书中用详细的案例展示了如何在实际项目中使用 Mock 对象：先设计出类之间的接口，然后为每个接口创建 Mock，编写测试来验证类之间的协作而非它们的内部实现。

这一章对比了 EasyMock 和 JMock 两种 Mock 框架。EasyMock 通过 java.lang.reflect.Proxy 动态生成 Mock 对象，JMock 通过 CGLib 生成。两者 API 风格不同，但核心思想一致——录制预期调用、设置返回值、验证结果。

### 第八章 容器内测试

Mock 对象在很多场景下工作得很好，但有些场景 Mock 无法覆盖：容器管理的事务、安全认证、EJB 的生命周期回调……这些只有在真实容器里才会生效的行为，Mock 无能为力。

这一章引入了容器内测试（In-container testing）的概念：把组件部署到真实容器（如 Tomcat、JBoss）中再进行测试。这是一种集成测试，代价是运行速度慢、环境配置复杂，但能验证在 Mock 环境下无法验证的行为。

三种测试策略的对比：

- **Stub**：轻量、快，但只验证返回值，不验证交互
- **Mock**：轻量、快、验证交互，但依赖手工编写或框架生成 Mock 对象
- **容器内测试**：重量、慢，但能验证真实环境的完整行为

没有绝对最优的策略，只有在具体场景下更合适的选择。

## 第三部分：构建与 CI

### 第九章 从 Ant 运行 JUnit

这一章介绍 Apache Ant——当时主流的 Java 构建工具。

Ant 的核心概念是 Target（目标）、Task（任务）和 Property（属性）。build.xml 文件定义了项目如何被构建，其中< junit>任务和< batchtest>任务是最常用的 JUnit 集成方式。< junit>任务可以生成 XML 格式的测试报告，这个报告可以被 CI 服务器解析。

Ant 另一个有用的功能是通过 Ivy 进行依赖管理——指定 JAR 包的坐标，Ivy 自动从 Maven 仓库下载。

### 第十章 从 Maven2 运行 JUnit

Maven 是对 Ant 的另一种回答。如果说 Ant 是「灵活但你需要自己定义一切」，Maven 就是「约定优于配置」——你只需要把项目按 Maven 的约定布局，Maven 就能自动推断出构建流程。

这一章介绍了 Maven 的核心概念：POM（Project Object Model）、坐标系统（groupId、artifactId、version）、依赖作用域（test、compile、runtime）。JUnit 依赖在 Maven 里声明为 test 作用域，意味着它不会被打包到最终产物里。

maven-surefire-plugin 是执行测试的插件，它自动发现 src/test/java 下的 Test*.java 和 *Test.java 文件并运行。< surefire-report:report>目标生成 HTML 格式的测试报告。

书中也坦率地指出了 Maven 的缺点——学习曲线陡峭、插件质量参差不齐、POM 膨胀——这些评价在多年后的今天看来依然有效。

### 第十一章 持续集成工具

JUnit 配合 Ant 或 Maven 让测试可以自动化运行，但还需要一个触发器让这个过程在每次代码提交时自动执行。这就是 CI（持续集成）的价值。

这一章介绍了当时两个主流的 CI 服务器：CruiseControl 和 Hudson（后来变成了 Jenkins）。

CI 的基本流程是：代码提交到版本库 → CI 服务器检测到变化 → 触发 Ant 或 Maven 构建 → 运行 JUnit 测试 → 生成测试报告 → 通知团队成员。如果测试失败，后续的部署步骤自动中止。

持续集成的核心价值不是工具本身，而是一种开发纪律：保持代码库的可用状态、及早发现集成问题、让每个人对构建状态负责。

## 第四部分：Web 层测试

### 第十二章 表现层测试

这一章讨论 GUI 层测试，特别是 Web 应用的用户界面。

HtmlUnit 是一个纯 Java 的「无头」浏览器，不启动真实浏览器但在内存中模拟 HTML 解析和 JavaScript 执行。适合快速自动化测试，也适合 CI 环境。

Selenium（当时是 Selenium RC）启动真实浏览器（Firefox、Chrome 等），通过 JavaScript 注入控制浏览器行为。真实性更高，但运行速度慢、维护成本高。

两者选择的本质是「速度/稳定性」和「真实性/覆盖度」之间的权衡。HtmlUnit 适合日常开发中的快速回归，Selenium 适合最终的端到端验收测试。

### 第十三章 Ajax 测试

2007 年前后 Ajax 正火，这一章是第十二章的延续，专门讨论 Ajax 应用带来的测试挑战。

Ajax 应用的核心问题是：页面的部分内容在初始加载后还会异步变化，传统的「请求-响应」式测试不再够用。

作者提出了一种分层测试策略：

- **功能测试**：用 Selenium 测试完整的用户交互流程
- **JavaScript 单元测试**：用 RhinoUnit 或 JsUnit 测试 JS 逻辑
- **服务端测试**：用 HttpClient 测试 Ajax 调用的后端接口

这一章还讨论了 GWT（Google Web Toolkit）应用的测试。GWT 将 Java 编译成 JavaScript，测试挑战在于编译后代码的行为可能与原始 Java 代码有差异。

### 第十四章 服务端 Java 测试——Cactus

Cactus 是这一章的主角，它是一个容器内测试框架，专门用于测试 Java EE 服务端组件（Servlet、Filter、JSP、EJB）。

Cactus 的设计思路是：测试运行在容器内（in-container），而不是在容器外通过 Mock 模拟。这意味着你可以测试真实的容器行为——session 管理、过滤器链、生命周期回调。

Cactus 与 Ant 和 Maven 都能集成，通过 Cargo 库自动下载和启动嵌入式容器。执行< cactus>任务后，Cactus 会启动容器、部署 WAR、运行测试、收集结果、关闭容器。

这一章的内容和第八章形成呼应——第八章是概念层面讲容器内测试的优势和劣势，第十四章是具体到 Cactus 这个工具的实现细节。

### 第十五章 JSF 应用测试

JSF（JavaServer Faces）是当时 Java EE 的标准 UI 框架。这一章的核心问题是：如何测试 JSF 应用？

JSF 的挑战在于它的组件树模型——与传统 Servlet 不同，JSF 请求处理经过生命周期多个阶段（Restore View、Apply Request Values、Process Validations、Update Model Values、Invoke Application、Render Response），在每个阶段组件的状态都可能变化。

JSFUnit 是专门为 JSF 设计的测试工具，集成在 Cactus 生态里。它允许测试在 JSF 生命周期的特定阶段插入断言，验证组件树的状态、托管 Bean 的值、验证器的结果等。

### 第十六章 OSGi 组件测试

OSGi 是当时「模块化 Java」的标准方案，Equinox、Felix 等实现开始流行。

OSGi 的核心概念是 Bundle——比传统 JAR 更精细的模块单元。Bundle 有生命周期（Install、Start、Stop、Update、Uninstall），可以在运行时动态加载和卸载。

这一章引入了 JUnit4OSGi 框架，它让 JUnit 测试可以在 OSGi 容器内运行，从而测试 Bundle 之间的依赖关系和动态服务交互。测试 OSGi 服务需要解决服务注册、服务查找、服务属性验证等一系列与普通单元测试不同的问题。

## 读后感

读完这本书，最大的感受是它记录了一个特定的测试技术时代。Ant 和 Maven2 并行、Cactus 用于容器内测试、JSFUnit 解决 JSF 测试问题、OSGi 还是热点话题——这些都是 2007 年前后的主流技术。

但书中有两条主线贯穿始终，从未过时：

第一条是**隔离的哲学**。无论是 Stub、Mock 还是容器内测试，核心思想都是把被测代码和它的依赖分离。依赖越多，测试越难写；依赖越隐藏，测试越脆弱。所以「写可测试的代码」和「用正确的方式隔离依赖」是两件相辅相成的事。

第二条是**自动化 over 手工**。手工测试能发现问题，但无法规模化。Ant/Maven 让测试成为构建的一部分，CI 服务器让测试成为每次提交的一部分——一旦这套机制建立起来，回归 bug 的发现就从一个靠运气的概率事件变成了一个确定性的自动化过程。

JUnit 本身在这本书出版后继续演进，从 JUnit 4 走到了 JUnit 5，设计思路有了根本性的变化。如果有机会读第三版（Catalin Tudose，2025），会是理解 JUnit 5 Extension Model 和 Jupiter 引擎很好的起点。但作为理解单元测试思想和 JUnit 4 时代技术生态的窗口，Vincent Massol 的这一版依然值得一读。
