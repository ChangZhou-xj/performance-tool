# 读书笔记 | 《JavaScript测试驱动开发》

## 核心收获

### TDD 的节奏感——红、绿、重构

书里花了很多笔墨讲这个循环，我之前以为这只是 Kent Beck 那本书的内容，Venkat 把它讲得更具体：

> "先编写一个不会通过的测试，然后编写最少的代码使得它通过，最后通过重构让代码更加完善。"

Venkat 的讲解方式是用一个华氏度转摄氏度的 f2c 函数，一步一步带着做。过程是这样的：

1. 先写一个金丝雀测试（canary test），确保测试环境没问题
2. 写第一个测试：32°F → 0°C，测试失败（红）
3. f2c 函数只返回 0，测试通过（绿）
4. 再写第二个测试：50°F → 10°C，测试失败
5. 修改 f2c 实现为 `(fahrenheit - 32) * 5 / 9`，两个测试都通过
6. 检查代码，有无不妥，没有就不动

这个过程有意思的地方在于：每次只改最少的东西让测试通过，绝对不过度设计。Venkat 说了一句让我印象很深的话："测试驱使我们真正实现了这个方法。但它同时也带来了一些问题。"——因为每一步都有测试护着，你敢去想更极端的输入，敢重构，敢清理。

### 金丝雀测试

这个概念我之前没接触过，觉得挺实用：

金丝雀测试是最简单的测试，只验证测试环境能跑通。写一个 `expect(true).to.eql(true)`，通过了说明 Mocha + Chai + Karma 安装无误，Node 版本没问题。

Venkat 建议把这个金丝雀测试保留在测试文件里。以后换了电脑、更新了工具链、改了环境变量，这个测试第一个跑，能快速告诉你"环境还正常"。

### 3A 模式

每个测试都遵循 3A 节奏：

```
Arrange — 准备测试需要的数据
Act — 执行被测代码
Assert — 验证结果
```

三个部分之间用空行隔开。这个模式让测试读起来像三句话，很有顺序感。Venkat 还特别强调：如果测试代码很长，通常说明测试设计有问题，或者背后的代码设计有问题。

### 测试替身——fake、stub、mock、spy

这是第 4 章的内容，我觉得是全书最值钱的一章。Venkat 引用了 Martin Fowler 的区分：

- **fake**：有实际逻辑但不能上生产的实现。比如测试时用假的信用卡服务。
- **stub**：被调用时返回预设数据，用来验证依赖返回后代码的行为。不验证交互，只验证状态。
- **mock**：除了返回预设数据，还追踪调用次数、顺序，用来验证被测代码和依赖之间的交互。
- **spy**：代理真实对象，既能执行真实逻辑，也能记录和验证交互。

Venkat 举了一个定位功能的例子：测试 `onSuccess` 时，需要调用 `createURL` 和 `setLocation`，但不能真的发网络请求。用 mock 还是 spy 取决于你要验证的是"是否调用了"（mock/spy）还是"调用后状态是否正确"（stub）。

个人体会：之前我一直混用 mock 和 stub，看完这章才建立直觉——**stub 验证状态，mock 验证行为（交互）**。

### 依赖注入

Venkat 用了一个 window.location 的例子：如果函数内部直接引用 `window.location`，测试时根本没法注入假的 window 对象。但如果把 window 作为参数传进来，测试时就能传一个 stub。

```javascript
// 直接依赖，难以测试
this.setLocation = function(url) {
  window.location = url;
}

// 依赖注入，易于测试
this.setLocation = function(window, url) {
  window.location = url;
}
```

这个例子让我意识到：**很多"代码难测"的问题，根源不是测试写不好，是代码结构本身就不利测试**。TDD 逼你写出可测的代码。

### Sinon 的价值

手动写测试替身很快会变得繁琐。Sinon 的 sandbox 模式可以自动清理，每次测试完恢复原始函数，省去手动保存/还原的步骤。Venkat 给了一个具体建议：手动写测试替身不费劲的时候就不需要 Sinon；如果代码结构因为测试替身变复杂了，就用 Sinon。

### 测试金字塔

第 10 章讨论集成测试和 E2E 测试时提到了测试金字塔——底部是大量的单元测试，往上是集成测试，最顶端是少量的 E2E 测试。Venkat 的观点很明确：

> "集成测试和端到端测试是在应用的所有部分都启动和运行后才得以完成的。虽然提供了极大的信心，但它们缓慢且脆弱。"

他建议：编写最少且有针对性的集成测试，只覆盖没有被其他层次覆盖到的部分。**在能得到快速、有意义反馈的最低层进行测试**。

这个观点我后来反思自己之前写的一些测试，有些明明在单元层就能验证的东西，非要在 E2E 里做，白白拖慢了整个测试套件。

### TDD 改善设计

第 11 章是总结性的一章，Venkat 列举了糟糕设计带来的问题（长函数、低内聚、高耦合、重复代码），以及 TDD 如何自然地规避这些问题：

- 函数保持简短（书中提到最长的函数不超过 20 行）
- 高内聚、单一职责
- 松耦合
- DRY（Don't Repeat Yourself）

Venkat 提到，测试代码量大约是源码量的 3 倍。这不是目标，是自然结果。这个数字让我有点心理准备——做 TDD 要接受前期投入相当大。

## 书里写得不够详细的地方

1. **TDD 三定律**：书里没有明确写出 Kent Beck 的 TDD 三定律（先写失败测试、只写刚好通过的代码、先重构再写新测试），Venkat 的讲解是围绕 f2c 例子的实践展开的，但没有抽象成这三条规则。如果想看更纯粹的 TDD 哲学，需要参考 Kent Beck 的书。

2. **测试命名**：Venkat 强调测试名要清晰，但具体怎么命名没有展开。书里的测试名字都是 `'should pass if f2c returns 0C for 32F'` 这种陈述句，没有讨论命名约定的选择（比如 should / it / given-when-then 等不同风格）。

3. **覆盖率陷阱**：Venkat 提醒不要过于依赖覆盖率报告，但没有深入讨论哪些类型的代码容易被覆盖率误导（比如异常处理、分支条件）。实际项目中看到过覆盖率很高但漏测一堆 bug 的情况，这块希望能再深入一点。

## 实践方向

结合书里的内容和自己做绩效考核工具的经验，我想到几个可以 TDD 化的场景：

- 节假日判断逻辑（HolidayService）：有很多边界条件（闰年、工作日判断），适合用测试先行的方式写
- 数据导出 Excel 的格式校验：文件名、sheet 数量、列头格式，这些用测试固定下来比手动回归靠谱
- 日报/周报的生成逻辑：时间范围计算、聚合统计，用测试确保改动不破坏已有逻辑

目前项目里的 HolidayService 其实已经写过一些测试，但覆盖率还不完整。下次加新功能的时候试试先写测试。

---

## 一句话总结

这本书最大的价值不是教会我用 Mocha 或 Sinon，而是让我理解：**测试先行不只是质量保障手段，它是一种设计方法——让你从代码使用者的角度出发，倒逼出更好的接口、更松的耦合、更短的函数。**
