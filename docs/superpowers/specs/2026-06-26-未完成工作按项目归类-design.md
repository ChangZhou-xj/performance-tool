# 未完成工作按项目归类 — 设计规格

## 背景

日报「七、明日工作计划」与周报「未完成工作」章节当前将所有进行中任务（`nextPlanItems`）平铺为一个列表，不区分任务所属性质。

需求：将未完成工作按项目归类：
1. 项目名称能匹配到 [config/project-config.js](../../config/project-config.js) 中某个项目 → 归类为「项目对接」
2. 其余 → 归类为「开发工作」

## 关键决策（已与用户确认）

1. **匹配规则复用 `matchKeywords`，不做字面 `displayName` 包含。**
   - 真实 Excel「项目名称」列里，仅「福建电力职业技术学院2025年智慧财务软件实施项目」「国家体育总局反兴奋剂中心」恰好等于 displayName；而 displayName「辽宁沈阳大学项目」「云南个旧项目」是人工简称，Excel 实际值为「辽宁省沈阳大学智慧财务一体化实施服务项目」「个旧市智能支出服务与监督管理平台实施项目」，不会被字面包含。
   - 现有 [generate-project-email.js](../../generate-project-email.js) 的 `matchProject` 已用 `matchKeywords`（如「沈阳大学」「个旧」「福建电力」「反兴奋剂/体育总局」），四个对接项目均能可靠命中。复用此逻辑，口径统一、归类准确。

2. **作用范围：日报 + 周报都改。**
   - `nextPlanItems` 同时出现在日报「七、明日工作计划」与周报「未完成工作」，两处一并归类，逻辑统一。

3. **展示格式：分「项目对接」「开发工作」两个子标题，各自从 1 编号。**
   - 组内仍按计划完成时间排序（`sortByPlannedFinishPriority`）、保留 A8 处理人信息追加。
   - 空组省略子标题；两组都空时输出「暂无」，保持原行为。

## 数据流

`nextPlanItems` 的 item 对象由 `pushUnique` 生成，已包含 `projectName` 字段（见 [generate-report-kf.js:310-323](../../generate-report-kf.js#L310-L323)）。因此分类时直接读取 `item.projectName`，调用 `matchProject` 判定归属，无需改动数据抽取阶段。

## 实现方案

### 组件 1：共享匹配模块 `service/project-match.js`（新建）

原因：`matchProject` 当前定义在 [generate-project-email.js](../../generate-project-email.js)，而 [generate-report-kf.js](../../generate-report-kf.js) 被其 require，反向 require 会产生循环依赖。抽出为独立模块供双方复用，消除重复、统一口径。

**职责：** 输入项目名称字符串，返回匹配到的 project 配置对象或 `null`。

**接口：**
```js
// service/project-match.js
'use strict';
const projectConfig = require('../config/project-config');

/**
 * 根据项目名称匹配到项目配置
 * @param {string} projectName - Excel 中的项目名称
 * @returns {object|null} 匹配到的项目配置
 */
function matchProject(projectName) {
  const name = String(projectName || '');
  for (const project of projectConfig.projects) {
    if (project.matchKeywords.some((kw) => name.includes(kw))) {
      return project;
    }
  }
  return null;
}

module.exports = { matchProject };
```

**改动：**
- [generate-project-email.js](../../generate-project-email.js)：删除本地 `matchProject` 定义（第 25-37 行），改为 `const { matchProject } = require('./service/project-match')`。保留对 `project-config` 的 require（仍需 `projects` 拼收件人、`emailSubject`）。
- `module.exports` 中**保留** `matchProject` 条目（re-export 指向新模块的实现），保持向后兼容——现有调用方 `extractProjectData`（[generate-project-email.js:125](../../generate-project-email.js#L125)）及其他潜在依赖不受影响。

**对外行为：** 不变。逻辑与原 `matchProject` 一字不差，仅对空值做了 `String(projectName || '')` 防御。

### 组件 2：分组工具 `groupNextPlanItems`（新增于 generate-report-kf.js）

**职责：** 将 nextPlanItems 列表按项目归类拆为两组。

**接口：**
```js
/**
 * 将未完成工作按项目归类拆为「项目对接」与「开发工作」两组
 * @param {Array} items - nextPlanItems（已 collectUniqueItems 去重）
 * @returns {{ dockingItems: Array, devItems: Array }}
 */
function groupNextPlanItems(items) {
  const dockingItems = [];
  const devItems = [];
  for (const item of items) {
    if (matchProject(item.projectName)) {
      dockingItems.push(item);
    } else {
      devItems.push(item);
    }
  }
  return { dockingItems, devItems };
}
```

复用现有 `buildSectionList`：各组分别调用，自带 `sortByPlannedFinishPriority` 排序、从 1 独立编号、A8 处理人追加（a8InfoMap）。

### 组件 3：日报展示（`buildReportMarkdown`）

将原：
```js
const nextPlanItems = collectUniqueItems(reportData.nextPlanItems);
// ...
markdown += `七、${nextPlanLabel}工作计划：\n`;
markdown += buildSectionList(nextPlanItems, '    ', sortByPlannedFinishPriority, a8InfoMap);
```
改为：
```js
const nextPlanItems = collectUniqueItems(reportData.nextPlanItems);
const { dockingItems, devItems } = groupNextPlanItems(nextPlanItems);
// ...
markdown += `七、${nextPlanLabel}工作计划：\n`;
if (dockingItems.length === 0 && devItems.length === 0) {
  markdown += `    暂无\n`;
} else {
  if (dockingItems.length > 0) {
    markdown += `    项目对接：\n`;
    markdown += buildSectionList(dockingItems, '      ', sortByPlannedFinishPriority, a8InfoMap);
  }
  if (devItems.length > 0) {
    markdown += `    开发工作：\n`;
    markdown += buildSectionList(devItems, '      ', sortByPlannedFinishPriority, a8InfoMap);
  }
}
```

缩进对齐现有「一、今日工作内容」结构（4 空格子标题、6 空格列表项）。

## 错误处理

- `matchProject(null)` / `matchProject('')` → `null`，归入「开发工作」，符合预期。
- 无新增错误路径；`buildSectionList` 已处理空列表（返回 `暂无`，但本设计在分组层即对空组省略）。
